Node.js es el terreno donde construimos.
Express es la estructura de la casa.
dotenv es el archivo donde guardamos llaves privadas.
pg es el puente hacia PostgreSQL.
Pool es el administrador de conexiones a la base de datos.
nodemon es el ayudante que reinicia el servidor cuando haces cambios.

Express es una librería/framework para crear servidores web y APIs con Node.js.
Cliente → petición HTTP → servidor Express → respuesta HTTP

dotenv sirve para cargar variables de configuración desde un archivo .env.

pg es la librería que permite conectar Node.js con PostgreSQL.
También se conoce como node-postgres.
Sin pg, Express puede recibir peticiones, pero no sabría cómo hablar con la base de datos PostgreSQL.

| Herramienta | ¿Qué es?                       | ¿Para qué sirve en IntelliPark?            |
| ----------- | ------------------------------ | ------------------------------------------ |
| Express     | Framework web                  | Crear la API REST del parqueadero          |
| dotenv      | Gestor de variables de entorno | Guardar puerto, credenciales DB y secretos |
| pg          | Librería de PostgreSQL         | Conectar Node.js con la base de datos      |
| Pool        | Administrador de conexiones    | Reutilizar conexiones a PostgreSQL         |
| nodemon     | Herramienta de desarrollo      | Reiniciar el servidor automáticamente      |

| Archivo                  | Responsabilidad                                                                                                |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------|
| `package.json`           | Define dependencias, scripts y configuración básica del proyecto.                                              |
| `.env`                   | Guarda variables sensibles o configurables: puerto, credenciales de base de datos, JWT secret, Key sonarqube   |
| `.gitignore`             | Evita subir `node_modules` y `.env` al repositorio.                                                            |
| `src/app.js`             | Configura Express, middlewares y rutas base.                                                                   |
| `src/server.js`          | Levanta el servidor en el puerto definido.                                                                     |
| `src/config/database.js` | Centraliza la conexión a PostgreSQL usando `pg`.                                                               |

======== Ahora falta completar el ciclo operativo del parqueadero. ========

Lista de lo que falta para terminar el backend MVP
1. Módulo de salida de vehículos

Este es el siguiente bloque obligatorio.

Endpoint sugerido:

POST /api/parking/check-out

Debe permitir:

- Buscar un vehículo por placa.
- Validar que tenga un ingreso activo.
- Calcular cuánto tiempo estuvo en el parqueadero.
- Calcular el valor a pagar. (La hora completa)
- Actualizar el registro como finalizado.
- Guardar hora de salida, minutos totales y valor total.

Regla principal:

Un vehículo solo puede salir si tiene un ingreso activo.

2. Definir tarifa del parqueadero

Antes o durante el check-out necesitamos decidir cómo se cobra.

Para una primera versión simple:

Carro: $3.000 por hora o fracción
Moto: $2.000 por hora o fracción

Podemos manejarlo inicialmente en código dentro del servicio, sin crear tabla de tarifas todavía.

Más adelante, si quieres algo más profesional, se crea una tabla:

parking_rates

Empezar con tarifa fija en el servicio.

3. Módulo de pagos

Después de calcular el valor, necesitamos registrar el pago.

Endpoint sugerido:

POST /api/payments
GET /api/payments
GET /api/payments/:id

Debe permitir:

- Registrar pago de un parqueo finalizado.
- Validar que el parqueo exista.
- Validar que no tenga pago registrado previamente.
- Guardar método de pago.
- Guardar valor pagado.
- Guardar usuario que registró el pago.

Regla principal:

No permitir pago duplicado para el mismo registro de parqueo.

Métodos de pago iniciales:

efectivo
transferencia
tarjeta
otro

4. Módulo de reportes básicos

Los reportes deben basarse en pagos registrados, no solo en parqueos finalizados.

Endpoints sugeridos:

GET /api/reports/income/daily
GET /api/reports/income/weekly
GET /api/reports/income/monthly

O una opción más flexible:

GET /api/reports/income?from=2026-05-01&to=2026-05-31

Debe permitir consultar:

- Total de ingresos.
- Cantidad de pagos.
- Total por método de pago.
- Total por tipo de vehículo.

Regla principal:

Los reportes financieros deben salir desde la tabla payments.
