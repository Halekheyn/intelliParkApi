Bloque 11 — Instalación y configuración de Swagger

Este bloque permite incorporar Swagger al backend de IntelliPark para documentar, visualizar y probar los endpoints de la API desde el navegador.

Swagger funciona como una documentación interactiva de la API. Permite ver las rutas disponibles, los métodos HTTP, los datos que se deben enviar, las respuestas esperadas y también probar endpoints protegidos con JWT.

A. Objetivo

Dejar disponible la documentación interactiva de la API en:

GET http://localhost:3000/api/docs

Y dejar disponible el archivo JSON de OpenAPI en:

GET http://localhost:3000/api/docs.json

Esto permite que el backend tenga una documentación más clara para pruebas, revisión técnica y futura conexión con el frontend.

B. Dependencias instaladas

Se instalaron las siguientes dependencias:

swagger-ui-express
swagger-jsdoc

swagger-ui-express permite mostrar la documentación de Swagger en una interfaz web.

swagger-jsdoc permite construir la documentación OpenAPI a partir de comentarios escritos dentro de los archivos de rutas.

C. Archivo de configuración

Se creó el archivo:

src/config/swagger.js

Este archivo contiene la configuración principal de Swagger/OpenAPI para el proyecto IntelliPark.

En este archivo se definieron elementos como:

- Nombre de la API.
- Versión de la API.
- Descripción general del proyecto.
- Servidor local de desarrollo.
- Tags o categorías de endpoints.
- Esquema de autenticación con Bearer Token.
- Esquemas reutilizables para los cuerpos de las peticiones.
- Ubicación de los archivos donde Swagger debe leer la documentación.

D. Ruta de documentación

Se configuró Swagger en app.js para que la documentación quede disponible en:

/api/docs

Desde esta ruta se puede abrir Swagger UI en el navegador y consultar la documentación interactiva de la API.

E. Ruta del JSON OpenAPI

También se configuró una ruta para exponer el archivo JSON de la especificación OpenAPI:

/api/docs.json

Esta ruta permite ver la estructura técnica completa de la documentación generada.

F. Integración en app.js

En el archivo app.js se agregaron las configuraciones necesarias para conectar Swagger con Express.

La documentación de Swagger debe registrarse antes del middleware de rutas no encontradas.

Si Swagger se registra después del notFoundHandler, la ruta /api/docs podría responder como si no existiera.

G. Organización por tags

Los endpoints se organizaron por categorías para facilitar la lectura:

Verificación del estado de la API.

Registro, login y consulta del usuario autenticado.

Users:
Gestión de usuarios.

Registro y consulta de vehículos.

Ingreso y salida de vehículos del parqueadero.

Registro y consulta de pagos.

Reportes básicos de ingresos.

H. Autenticación en Swagger

Se configuró un esquema de seguridad llamado bearerAuth.

Este esquema permite probar endpoints protegidos con JWT desde Swagger.

Para usarlo, primero se debe hacer login y copiar el token generado por el backend.

Luego, en Swagger, se debe hacer clic en el botón Authorize y pegar el token.

Después de autorizar, Swagger enviará el token en las peticiones protegidas mediante el header Authorization.

I. Importancia del Bearer Token

Los endpoints protegidos no reciben el userId desde el body, params o query.

El usuario autenticado se identifica a partir del token JWT.

Cuando el backend recibe una petición protegida, el middleware de autenticación valida el token y extrae de él los datos del usuario, como:

- userId
- email
- role

Por esta razón, rutas como:

GET /api/auth/user-me

no necesitan recibir un userId como parámetro.

El userId se obtiene directamente desde el JWT.

J. Documentación de rutas

Para que Swagger muestre los endpoints, se deben agregar comentarios de documentación en los archivos de rutas.

Estos comentarios indican:

- Método HTTP.
- URL del endpoint.
- Descripción de la operación.
- Categoría o tag.
- Si requiere autenticación.
- Body esperado.
- Parámetros.
- Respuestas posibles.

Los archivos principales donde se documentan rutas son:

src/routes/auth.routes.js
src/routes/users.routes.js
src/routes/vehicles.routes.js
src/routes/parking.routes.js
src/routes/payments.routes.js
src/routes/reports.routes.js

K. Prueba inicial

Después de instalar y configurar Swagger, se debe iniciar el servidor con:

npm run dev

Luego se abre en el navegador:

http://localhost:3000/api/docs

Si la configuración es correcta, Swagger debe mostrar la documentación interactiva de IntelliPark API.

También se puede probar:

http://localhost:3000/api/docs.json

Esta ruta debe mostrar la especificación OpenAPI en formato JSON.

L. Error común encontrado

Durante la prueba del endpoint de usuario autenticado, Swagger enviaba la petición sin el header Authorization.

Por esta razón el backend respondía:

El token de autorización es requerido.

Esto ocurrió porque el endpoint protegido no estaba enviando el JWT.

La solución fue usar el botón Authorize de Swagger y asegurarse de que la ruta estuviera documentada con seguridad bearerAuth.

Swagger se incorporó como herramienta de documentación y prueba del backend.

Esto permitirá que el proyecto IntelliPark tenga una documentación más profesional, útil para pruebas con aprendices, revisión técnica y futura integración con Angular.
